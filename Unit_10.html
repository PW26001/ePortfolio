Unit 10: Test-Driven Development (TDD) and Behaviour Driven Development (BDD)

TDD = Fail → Pass → Improve.

Fail = write the expected behaviour as a test before the code works.
Pass = write enough code to meet that expectation.
Improve = refactor the code while keeping the test passing.

As I do not come from a development background, the terminology used here was new to me. While I may have inadvertently approached certain tasks in a similar way, the formality of the TDD approach is a new way of thinking and something I can envision applying to my professional life.

Whilst TDD was new, Unit Testing feels like an approach I've definitely taken, and through the wider reading I realised how closely Dependency Injection supports Unit Testing by allowing parts of a system to be tested in isolation.

Checking a small part of the code in isolation before trusting it in the wider system felt very natural, even if the description here was for much more complex scenarios than I am used to facing. Proving expected behaviour, and catching mistakes while the specific portion I am working on is still small, was easy to correlate with my own experience.

I also enjoyed recognising the natural meeting point of this subject with earlier units, where TDD ensures that refactoring does not break existing functionality. This locked in a lot of the theory for me.
