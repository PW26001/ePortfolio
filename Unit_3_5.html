Units 3-5: Design Patterns

Units 3–5 focused on design patterns, covering creational, structural and behavioural patterns. I found these units useful because they helped me move from understanding individual SOLID principles to seeing reusable ways those principles can be applied in code.

The Factory Method task showed how object creation can be moved away from the main program, allowing new car types to be introduced without changing the client code. The structural pattern examples helped me understand how patterns such as Adapter, Bridge and Composite solve problems around compatibility, separation and hierarchy.

The Strategy Pattern task was particularly useful because it showed how long if/elif decision blocks can be replaced with interchangeable behaviours.

The artefacts show my developing understanding that design patterns are not just academic examples. They are ways of making software easier to extend, test and maintain.

I appreciated the structure that these modules introduced, and at this point it influenced my working life. I was encouraged by this new knowledge to go away and create reusable PowerShell modules which would improve the process that the team follow when migrating between deployment systems, taking care to ensure that all stages were clearly implementing abstraction to allow for updates in future, while using a consistent reusable structure which would allow for expansion when required.

The learning material repeatedly explained that unnecessary patterns could lead to overcomplication of simple programs and I found this to be true, often expanding and then contracting the modules I was creating to only implement the features that were absolutely needed.
