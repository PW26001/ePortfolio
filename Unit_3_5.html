Units 3-5: Design Patterns

Units 3–5 focused on design patterns, covering creational, structural and behavioural patterns. I found these units useful because they helped me move from understanding individual SOLID principles to seeing reusable ways those principles can be applied in code.

Unit 3 introduced creational design patterns. The Factory Method task showed how object creation can be moved away from the main program, allowing new car types to be introduced without changing the client code. I found this useful because it linked clearly to my earlier understanding of separating responsibilities and avoiding hardcoded logic.

Unit 4 covered structural design patterns. The Adapter, Bridge and Composite examples helped me understand how patterns can solve problems around compatibility, separation and hierarchy. I found it useful to think of Adapter as connectivity between non-compatible interfaces, Bridge as separating independent aspects of an object, and Composite as organising objects into a tree structure so individual items and collections can be treated the same way.

Unit 5 covered behavioural design patterns. The Strategy Pattern task was particularly useful because it showed how long if/elif decision blocks can be replaced with interchangeable behaviours. This helped me understand how behaviour can be passed into an object rather than being fixed inside it.

The Cupboard Sorter artefact builds on my earlier Unit 2 work. Rather than forcing every pattern into the example, I applied the patterns that fitted most naturally: Factory Method, Decorator, Strategy and Chain of Responsibility. This helped show how the same basic example could develop from SOLID principles into design-pattern thinking.

The artefacts show my developing understanding that design patterns are not just academic examples. They are ways of making software easier to extend, test and maintain.

I appreciated the structure that these modules introduced, and at this point it influenced my working life. I was encouraged by this new knowledge to go away and create reusable PowerShell modules which would improve the process that the team follow when migrating between deployment systems, taking care to ensure that all stages were clearly implementing abstraction to allow for updates in future, while using a consistent reusable structure which would allow for expansion when required.

The learning material repeatedly explained that unnecessary patterns could lead to overcomplication of simple programs and I found this to be true, often expanding and then contracting the modules I was creating to only implement the features that were absolutely needed.
